# Концепция sorm

## Позиционирование

**«EF Core для Go»: первая Go ORM с честным Unit of Work.**

Загрузил граф сущностей → мутировал структуры обычным Go-кодом → `session.SaveChanges(ctx)` → минимальный дифф, упорядоченный по FK-зависимостям, ушёл в БД одним roundtrip'ом. Ни одна из ~десяти заметных Go-библиотек этого не делает ([gap-анализ](efcore-gap-analysis.md), разрыв №2) — при этом всё необходимое (generics, кодоген, `pgx.Batch`) в Go 1.23+ есть.

Целевые СУБД: **PostgreSQL (pgx, приоритет), MySQL, SQLite.**

Почему сейчас удачный момент:
- GORM своим generics API (2025) фактически признал дефекты старого дизайна, но не может сломать совместимость и не имеет UoW.
- Консенсус «sqlc + pgx» трещит на динамических запросах — стандартный ответ «две системы запросов в одном кодбейзе».
- sqlx заморожен, SQLBoiler в maintenance mode — установленная база ищет, куда мигрировать.
- Ниша «типобезопасная ORM с change tracking» пуста во всей экосистеме.

## Три кита

### 1. Сессия с change tracking (ключевой дифференциатор)

Механика — snapshot-дифф, как дефолтный трекер самого EF Core (прокси не нужны):

- Кодоген выдаёт на каждую сущность snapshot-структуру и пополевой `Diff(old)` — **ноль рефлексии в рантайме**.
- Сессия держит identity map (`map[PK]*T`): повторная загрузка той же строки возвращает тот же указатель.
- `SaveChanges(ctx)`: дифф всех отслеживаемых сущностей → топологическая сортировка по FK-графу (известен на этапе кодогена) → гетерогенный батч insert/update/delete → один roundtrip (`pgx.Batch`; на MySQL/SQLite — multi-statement в транзакции).
- **Tracking — opt-in.** Запросы по умолчанию не отслеживаются (AsNoTracking-семантика): дефолт лучше, чем у EF, и честнее для Go-культуры явности.

Бесплатные следствия трекера:
- **Optimistic concurrency**: поле-версия в схеме → каждый генерируемый UPDATE несёт `WHERE version = ?` + проверку rows-affected; конфликт → типизированная ошибка `ConflictError` со значениями из БД.
- **Частичные апдейты без ручных списков полей** — устраняет целый класс багов GORM (`Updates` c zero-values) и sqlc (полная перезапись).

### 2. Типобезопасные запросы (аналог LINQ через кодоген)

Генерируемый DSL уровня Jet/Ent: типизированные колонки, предикаты-значения, композиция условий обычным Go-кодом — закрывает ахиллесову пяту sqlc (динамические фильтры) без второго query builder'а.

Отличия от существующих:
- **Компактный сгенерированный код** (анти-Ent): плоские таблицы метаданных + generic-рантайм вместо тысяч строк на сущность. Цель — на порядок меньше кода, чем у Ent, при той же типобезопасности.
- **Нативный pgx**: сканирование (в т.ч. вложенные структуры из JOIN'ов) работает с pgx, а не только с `database/sql` — устраняет главный практический минус Jet.
- **Предсказуемый SQL**: генерируем ровно тот запрос, который ожидает SQL-грамотный разработчик (анти-Ent №2); `.ToSQL()` на любом запросе для инспекции.
- Eager loading с выбором стратегии: `Include(...)` по умолчанию split query (как Ent), опционально JOIN — настройка, которой нет почти нигде.
- Лёгкий escape: `sorm.Raw[User](ctx, db, "SELECT ...", args...)` — raw SQL с маппингом в те же модели и вложенным сканированием.

### 3. Миграции по диффу модели

Не изобретать движок диффа — **интегрировать Atlas** (лучший в экосистеме, местами превосходит `dotnet ef migrations add`: lint, drift detection, dev-database). sorm генерирует из schema-as-code каноническое описание схемы, которое Atlas диффует в версионные SQL-миграции. Собственный CLI `sorm migrate diff` — тонкая обёртка. Если привязка к Atlas Cloud станет проблемой — план Б: собственный дифф для подмножества DDL (таблицы/колонки/индексы/FK) по трём целевым диалектам.

## DX-принципы (выведены из чужих ошибок)

| Принцип | Чей провал закрываем |
|---|---|
| Никакого тихого отбрасывания zero-values: условия и апдейты — только через типизированные предикаты/дифф | GORM `Where(&User{Age: 0})`, `Updates(User{Active: false})` |
| Ошибки — только возвращаемые значения; не найдено → `ErrNotFound` единообразно во всех методах | GORM chained `.Error`, разнобой First/Find |
| `ctx` — обязательный первый аргумент всех операций с БД | GORM опциональный `WithContext` |
| Билдеры иммутабельны: каждое условие возвращает новое значение, state-leak невозможен | GORM SQL pollution |
| Вложенное сканирование JOIN'ов во структуры-графы из коробки, включая NULL в LEFT JOIN | sqlx issues #112/#162, sqlc `sqlc.embed()` |
| has-many: и split query, и JOIN-стратегия, фильтрация родителя по детям | Bun issue #604 |
| Никаких неявных глобальных фильтров без объявления в схеме; soft delete виден в типе запроса | GORM `DeletedAt` |
| Миграции версионные с первого дня; никакого AutoMigrate в проде | GORM AutoMigrate |
| Сгенерированный код компактен и читаем в PR | Ent |

## Архитектура

```
schema-as-code (Go)  →  sorm gen  →  генерируемый слой         →  тонкий generic-рантайм
  описание сущностей       CLI         • метаданные таблиц/FK        • Session / tracker
  полей, связей,                       • типизированные колонки      • query executor
  индексов, версий                       и предикаты                 • диалекты: pgx / MySQL / SQLite
                                       • Snapshot / Diff             • батчер (pgx.Batch / multi-stmt)
                                       • сканеры (в т.ч. вложенные)  • hooks / interceptors / OTel
```

Источник истины — схема в Go (путь Ent, а не database-first, как Jet/SQLBoiler: не требует живой БД для кодогена, работает в CI, схема проходит code review).

## Эскиз публичного API

```go
// --- схема (schema/user.go) ---
type User struct {
    ID      int64  `sorm:"pk,auto"`
    Email   string `sorm:"unique"`
    Active  bool
    Version int64  `sorm:"version"` // optimistic concurrency
    Posts   []Post `sorm:"hasMany:AuthorID"`
}

// --- запросы: генерируемый DSL, условия — значения ---
q := sorm.Query[User](db).
    Where(u.Active.Eq(true)).            // bool=false тоже работает — предикаты не знают zero-value
    With(u.Posts.Include().Join())       // eager: JOIN или split (дефолт) — явный выбор
if filter.MinAge != nil {
    q = q.Where(u.Age.Gte(*filter.MinAge)) // динамическая композиция — то, что не может sqlc
}
users, err := q.All(ctx)               // без tracking'а (дефолт)

// --- сессия: Unit of Work, tracking opt-in ---
s := sorm.NewSession(db)
user, err := sorm.Track[User](s).Where(u.ID.Eq(42)).One(ctx)
user.Active = false                     // обычная мутация структуры
user.Posts[0].Title = "новый заголовок"
err = s.SaveChanges(ctx)                // дифф → топосорт → один roundtrip
var conflict *sorm.ConflictError        // rows-affected ≠ ожидаемому при version-предикате
if errors.As(err, &conflict) { /* разрешение конфликта */ }
```

Детальный дизайн API, кодогенерации и алгоритма flush — в [design.md](design.md).

## Roadmap

1. **MVP (PG-only):** schema-as-code + `sorm gen` (метаданные, предикаты, snapshot/diff, сканеры) + query DSL + Session/SaveChanges поверх pgx с `pgx.Batch`. Это минимум, демонстрирующий дифференциатор.
2. **Мультидиалект:** абстракция диалекта, MySQL и SQLite; гетерогенный батч через multi-statement транзакции.
3. **Миграции:** интеграция с Atlas (`sorm migrate diff/apply`), экспорт канонической схемы.
4. **Зрелость:** optimistic concurrency полировка, retry execution strategies (`RunInTx` с классификацией transient-ошибок), hooks/interceptors, OpenTelemetry, типизированные JSON-предикаты.

Вне скоупа надолго: lazy-прокси (невозможно в Go), трансляция произвольного Go-кода в SQL, temporal tables.

## Риски

- **Кодоген — трение**: обязательный шаг сборки; смягчение — `go:generate`, быстрый инкрементальный генератор, компактный вывод.
- **Snapshot-дифф стоит памяти** (копия каждой tracked-сущности): именно поэтому tracking opt-in; для read-путей стоимость нулевая.
- **Культурное сопротивление** («в Go не нужны ORM»): позиционирование не «спрячь SQL», а «убери ручную бухгалтерию изменений»; предсказуемый SQL и `.ToSQL()` — ответ скептикам.
- **Atlas-зависимость** в миграциях: изолировать за интерфейсом, план Б описан выше.
