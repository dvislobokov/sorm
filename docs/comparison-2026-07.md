# sorm против экосистемы Go ORM — детальный анализ (июль 2026)

Состояние sorm: main @ PR 36 (v0.5.0+): к моменту обновления закрыты пункты 1–5 приоритетов ниже (MIT, upsert, подзапросы+ForUpdate, soft delete, entity-хуки), добавлены sormtest, WithReplicas, MariaDB/CockroachDB.
Сравниваем с: **GORM** (самый популярный), **Ent** (Meta/atlasgo, кодоген-граф),
**Bun** (uptrace, SQL-first builder), **sqlc** (SQL→Go генератор),
**sqlx** (тонкая обёртка над database/sql). Базовая линия — сырой pgx.

## 2. Где sorm объективно лучше всех

### 2.1 Настоящий Unit of Work — категория из одного участника
Ни один Go ORM не делает identity map + снапшот-дифф + автоматический
упорядоченный flush. У всех остальных «обнови сущность» = пользователь сам
вызывает Save/Update и сам следит, что изменилось:

- GORM `Save(&user)` пишет **все** колонки (или Updates с ручным перечислением);
- Ent требует явные мутации `user.Update().SetName(...)`;
- Bun/sqlc — руками.

sorm: загрузил → мутировал Go-кодом → `SaveChanges` — UPDATE только изменённых
колонок, вставки топологически отсортированы по FK-графу (включая
самоссылающиеся таблицы), детям проставляются ключи родителей, всё в одной
транзакции и минимуме roundtrip'ов (pgx.Batch). Дифф — генерированный код:
0 аллокаций, ~26нс на сущность, рефлексии в горячем пути нет вообще.

Плюс генерируемый `Context` (DbContext): tracked-сеты, `Find` через identity
map, `NoTracking`, `RunInTx` со свежим дочерним контекстом на попытку ретрая.
Это дословно EF Core-опыт, которого в Go не было.

### 2.2 Философия «ошибка сборки вместо тихо неверного SQL»
Систематическое преимущество, а не фича:

- zero values — полноценные условия (`Active.Eq(false)` работает; в GORM
  struct-условие с false молча исчезает — ловушка №1 по количеству инцидентов);
- `Update`/`Delete` без Where — ошибка, вся таблица только явным `AllRows()`
  (в GORM — `ErrMissingWhereClause` появился после лет глобальных апдейтов);
- диалект-специфика (PG-массивы, `pgagg`/`myagg`, JSON `Contains` на SQLite)
  на чужом диалекте — ошибка построения с внятным текстом, до БД не доходит;
- строгий маппинг проекций/Raw: лишняя или недостающая колонка — `*ScanError`
  заранее, а не полупустая структура.

### 2.3 Миграции: самый полный встроенный набор
Дифф-движок (Atlas SDK) внутри библиотеки — без внешнего CLI и docker-магии;
декларативный Apply/Plan **и** версионные файлы с down-скриптами, SHA-256
чексуммами каталога, advisory-lock'ом на репликах и seed-механикой
(`migrate.Seed` — one-time, транзакционно, в той же истории). GORM AutoMigrate
только добавляет колонки (не удаляет, не меняет типы, дрейф не видит).
Ent использует тот же Atlas, но тянет его как отдельный инструмент/сервис;
у нас — одна go-зависимость. Bun/sqlc миграций по сути не имеют.

### 2.4 Типизированный JSON глубже, чем у всех
`sorm:"json"` на структуре генерирует аксессоры по её форме:
`u.PrefsDoc.Notify.Push.IsTrue()` — с диалектно-правильными кастами
(PG `::boolean`, MySQL JSON_EXTRACT, SQLite 1/0). Ent/sqljson ближе всех, но
там пути — строки; GORM datatypes.JSON нетипизирован полностью.

### 2.5 Наблюдаемость
otelsorm даёт не только спаны, но продуманный набор метрик: длительности по
semconv, размер батча, классы statement'ов, rows.returned, tx.duration,
tx.retries, пул — плюс `WithQueryName`/`.Named()` для маркировки запросов.
otelgorm — только спаны с SQL; у Bun хорошая, но менее глубокая интеграция.

### 2.6 Производительность при высокоуровневости
Наши бенчмарки (benchmarks/, PG 17, одна машина):

| Сценарий | raw pgx | sorm | Ent | GORM |
|---|---|---|---|---|
| Query 1000 строк | 517µs | **565µs** (+9%) | 706µs (+37%) | 1039µs (+101%) |
| UpdateOne | — | **7.3µs** (быстрее всех) | ~9µs | ~15µs |
| Insert100 | — | паритет с GORM | — | паритет |

То есть цена UoW и типизации — ~9% к сырому драйверу; GORM платит 2×.

### 2.7 Мелочи, которые складываются
Мультитенантность одной обёрткой (`InSchema`), naming-стратегии генератора,
авто-таймстампы с честной семантикой (no-op flush не трогает updated_at),
uuid по значению, nullable-FK навигации, стриминг `Iter`, IN-чанкование,
расширяемые агрегаты (`NewAgg` + pgagg/myagg), `RunInTx` с классификацией
transient-ошибок по диалектам и джиттер-ретраями.

## 3. Где sorm хуже — честно

### 3.1 Зрелость и экосистема (главный разрыв)
- Один мейнтейнер, нулевой продакшен-пробег, нет community, вопросов на SO,
  готовых рецептов. GORM/Ent — годы продакшена, тысячи issue-прецедентов.
- ~~Нет LICENSE~~ — закрыто: MIT (v0.3.0).
- Нет v1.0, API не заморожен; нет сайта документации (только markdown в репо).

### 3.2 Функциональные дыры против конкретных конкурентов
**Против Ent:**
- граф-обходы (`user.QueryGroups().QueryUsers()...`) — наш Include/Any слабее
  для глубоких графовых запросов;
- privacy layer (политики доступа на уровне схемы), хуки/интерсепторы на
  мутациях, mixins/annotations, валидаторы полей;
- интеграции из коробки: GraphQL (gqlgen), gRPC, OpenAPI-генерация.

**Против GORM:**
- диалекты: нет SQL Server, ClickHouse, Oracle (у GORM есть официальные и
  community-драйверы);
- ~~dbresolver~~ — закрыто: sorm.WithReplicas (v0.5.0+); шардинг-плагинов нет;
- ~~soft delete из коробки~~ — закрыто: тег softDelete (v0.4.0);
- ~~хуки жизненного цикла~~ — закрыто: BeforeSave/AfterLoad (v0.4.0);
- prometheus/encryption-плагины — нет (OTel-экспортёр покрывает метрики);
- Scopes (переиспользуемые именованные куски запросов) — у нас композиция
  билдеров, но нет идиомы «глобальных скоупов».

**Против sqlc:**
- sqlc проверяет SQL против реальной схемы на этапе генерации — «SQL как
  истина» для команд с сильными DBA мы не заменяем;
- нулевой оверхед по определению.

**Против Bun:**
- у Bun богаче SQL-поверхность билдера: CTE, subquery, UNION, VALUES,
  window-функции — наш билдер уже (см. 3.3).

### 3.3 Общие пробелы SQL-поверхности
- составные первичные ключи сущностей (join-таблицы умеем, сущности — нет);
- ~~подзапросы IN (SELECT)/скалярные~~ — закрыто (v0.4.0); коррелированные
  скаляры в проекциях — по-прежнему нет;
- CTE/`WITH RECURSIVE` (спроектирован рекурсивный обход self-FK);
- window-функции в проекциях;
- ~~FOR UPDATE/SKIP LOCKED~~ — закрыто (v0.4.0);
- ~~UPSERT~~ — закрыто (v0.4.0);
- произвольные global query filters (кроме soft delete) — нет;
- Attach/Update без предварительного SELECT (detached-сценарии PUT-API);
- UNION, DISTINCT ON, lateral join — только через Raw.

### 3.4 Технические долги
- Atlas зафиксирован на v0.32.0 (v1.x требует Go 1.26) — отстаём от
  апстрима дифф-движка;
- PG-массивы работают только через pgxd (database/sql путь требует pq.Array);
- `sorm gen` CLI не исполняет код моделей → не видит кастомные `Indexes()`
  (предупреждаем, но это ограничение);
- MySQL-миграции пофайлово нетранзакционны (implicit commit DDL — ограничение
  MySQL, но у конкурентов та же беда);
- ошибки MySQL классифицируем по тексту сообщений (нет импорта драйвера в
  ядре — цена обещания «ядро без драйверов»).

## 4. Итоговая позиция

**sorm сегодня — единственный Go ORM с EF Core-моделью работы**: UoW,
tracked-контекст, дифф-flush, встроенные миграции с даун/чексуммами/сидом,
типизированный JSON и честная диалектная дисциплина — при накладных ~9% к
сырому pgx. По этим осям мы впереди всех, местами — вне конкуренции.

**Проигрываем** в зрелости (лицензия! стабильность API, community), в ширине
SQL-поверхности (upsert, подзапросы, CTE, окна, блокировки), в экосистемных
интеграциях (GraphQL/gRPC у Ent, плагины GORM) и в количестве диалектов.

Приоритеты, если закрывать разрывы по соотношению ценность/стоимость:
1. **LICENSE + v0.3.0** — административно, но блокирует любое использование.
2. **UPSERT** — самый частый вопрос к любому ORM, у конкурентов есть.
3. **Подзапросы (InQuery/скалярные) + ForUpdate** — спроектированы.
4. **Soft delete / global filters + entity-хуки** — закрывает GORM-привычки.
5. Рекурсивный CTE по self-FK — дифференциатор, которого нет ни у кого.
