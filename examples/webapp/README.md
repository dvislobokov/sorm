# webapp — a todo API on sorm

A complete example: `net/http` handlers on top of sorm, PostgreSQL, three
versioned migrations generated by the sorm tool, a Dockerfile and Compose.

What it demonstrates:

- **Unit of Work in handlers** — `POST /tasks/{id}/toggle`: `Track` → a plain
  struct mutation → `SaveChanges`; a concurrent modification is caught by
  optimistic concurrency and turned into `409 Conflict`.
- **Eager loading** — `GET /users` returns users with their tasks
  (split query, no N+1).
- **Dynamic filter composition** — `GET /tasks?done=false&min_priority=2`
  builds the WHERE clause piece by piece; `done=false` is a first-class
  condition (the zero value is not dropped).
- **Migrations on startup** — the app itself applies pending files from
  `migrations/` (`migrate.Up`); the history lives in the `sorm_migrations`
  table.

## Running

```bash
cd examples/webapp
docker compose up --build
```

Compose brings up PostgreSQL (with a healthcheck) and the app; on startup
the app applies the three migrations and starts listening on `:8080`.

## Trying it out

```bash
curl -s -X POST localhost:8080/users -d '{"name":"Alice","email":"alice@example.com"}'
curl -s -X POST localhost:8080/tasks -d '{"user_id":1,"title":"write the README","priority":2}'
curl -s -X POST localhost:8080/tasks -d '{"user_id":1,"title":"drink coffee","priority":5}'

curl -s localhost:8080/users                          # users with their tasks
curl -s "localhost:8080/tasks?done=false&min_priority=3"

curl -s -X POST localhost:8080/tasks/1/toggle         # Unit of Work
curl -s "localhost:8080/tasks?done=true"
```

## How the migrations were generated

The files in `migrations/` are not handwritten: they are real schema diffs
taken at three steps of the schema's evolution. For a diff, sorm needs an
**empty scratch DB** (to replay the existing migrations + compare against the
models). Bring it up yourself and pass its DSN — sorm does not launch
anything for you:

```bash
docker run -d --name scratch -e POSTGRES_PASSWORD=postgres -p 15433:5432 postgres:17-alpine
```

Then it's a "change the models → generate a diff" loop (from the repository
root; each diff needs a fresh empty database — `CREATE DATABASE scratchN`):

```bash
# step 1: models.go contains only User
go run ./cmd/sorm migrate diff -dialect postgres \
  -dir examples/webapp/migrations \
  -dev-dsn 'postgres://postgres:postgres@localhost:15433/scratch1' \
  init ./examples/webapp/models
# → 20260711200554_init.sql (CREATE TABLE users)

# step 2: added Task (hasMany/belongsTo, FK)
# → 20260711200614_add_tasks.sql (CREATE TABLE tasks + FK)

# step 3: added the Task.Priority field
# → 20260711200639_add_priority.sql (ALTER TABLE tasks ADD COLUMN priority)
```

After changing the models, don't forget to regenerate the typed layer:

```bash
go run ./cmd/sorm gen ./examples/webapp/models   # or go generate ./...
```

The same operations are available from code (`sorm/migrate`): `Diff`, `Up`,
`Pending` — the app in `main.go` uses `migrate.Up`.

## Layout

```
webapp/
├── main.go            HTTP server: migrations on startup + handlers
├── models/models.go   the schema (source of truth)
├── models/sormgen/    generated by `sorm gen` (checked into VCS)
├── migrations/        generated by `sorm migrate diff` (checked into VCS)
├── Dockerfile         multi-stage: golang:1.25 → distroless
├── compose.yaml       postgres + app
└── README.md
```

## Evolving the schema further

1. Change `models/models.go`.
2. `go run ./cmd/sorm gen ./examples/webapp/models`.
3. `go run ./cmd/sorm migrate diff -dialect postgres -dev-dsn <empty scratch DB> <name> ./examples/webapp/models`.
4. Review the new `.sql` in a PR.
5. Deploy: the app applies it on startup (or `sorm migrate up -dsn <prod>`).
